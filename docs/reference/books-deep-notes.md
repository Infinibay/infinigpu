## 1. PCI device model: BARs, config space, mmap-ing device memory

_Ties to our vfio-user server: the emulated GPU it presents must expose exactly this shape — config space + BARs + MSI-X + bus-master DMA. This is the guest driver's view of our device._

- A PCI function has **at most six BARs**; each BAR advertises "how much memory it needs, as well as its type" (mem vs I/O) and the OS assigns the window (Madieu-Mastering, p526).
- Config space is **256 bytes on conventional PCI, 4 KB on PCIe** — "The PCIe extends this to 4 KB of register space" (Madieu-Mastering, p525). Header-type register: "type 0 means an endpoint device and type 1 means a bridge device" (Madieu-Mastering, p520).
- `struct pci_dev` carries `vendor`/`device`/`class`/`revision`/`hdr_type`/`irq`/`dma_mask` plus `msi_enabled`/`msix_enabled` bitfields; `enable_cnt` is nested so the device only truly disables "after all callers of pci_enable_device() have called pci_disable_device()" (Madieu-Mastering, p535).
- Match table is an array of `PCI_DEVICE(vendor,device)` entries terminated by `{ 0, }`, exported via `MODULE_DEVICE_TABLE` for udev autoload; `struct pci_driver` needs `.name`/`.id_table`/`.probe`/`.remove` and `id_table` "must be non-NULL for the probe to be called" (Madieu-Mastering, p536–538). Register with `pci_register_driver()` or the `module_pci_driver()` one-liner (Madieu-Mastering, p540).
- **Probe ordering (load-bearing):** `pci_enable_device()` (initializes mem + I/O BARs) → `pci_set_master()` — the latter "must be invoked if the device is intended to perform DMA operations" → map BARs (Madieu-Mastering, p541–542).
- Sized config accessors: `pci_read_config_byte/word/dword(dev, where, &val)`, where `where` is an offset like `PCI_REVISION_ID`; word/dword auto-handle "little endian format" (Madieu-Mastering, p543).
- BAR geometry: `pci_resource_start()`, `pci_resource_len()`, `pci_resource_flags()` — test `IORESOURCE_MEM` vs `IORESOURCE_IO` before mapping (Madieu-Mastering, p545).
- Mapping a BAR into kernel space: raw path is `request_mem_region()` then `ioremap()` → use `ioread32()/iowrite32()`; preferred path is `pci_request_regions()` + `pci_iomap(dev, bar, maxlen)` (pass `0` for full length), or `pci_ioremap_bar()` which "makes sure the BAR is actually a memory resource, not an I/O resource" (Madieu-Mastering, p546–547). ioremap KVA "is typically an address within the vmalloc region" and access must use `ioread[8|16|32]()`/`iowrite[8|16|32]()`, never a raw deref (Billimoria-P2, p143, p385).
- The physical RAM our device DMAs from lives inside the VMM process address space (see §7) — an out-of-process device model must map/share that region.

## 2. DMA: coherent/streaming, scatter-gather, zero-copy buffer sharing

_The mechanics behind guest-page DMA and the shared buffer we hand to the host GPU._

- Include `<linux/dma-mapping.h>`, then declare addressing width with `dma_set_mask(dev, DMA_BIT_MASK(32|64))` — needed "especially if the device can directly address 'consistent memory' … above 4 GB". Bus addresses are `dma_addr_t`, seen "through the lens of the bus controller and any intervening I/O memory management unit (IOMMU)" (Madieu-Mastering, p557).
- **Coherent** (`dma_alloc_coherent(dev, size, &dma_handle, gfp)` / `dma_free_coherent`): returns a CPU virtual address + output bus address, is "uncached (coherent) and unbuffered", is **guaranteed physically contiguous**, min one page (rounded to power-of-2 pages via `get_order`), and is implicitly bidirectional — "use it for buffers that last the lifetime of the device" (Madieu-Mastering, p558–559; Madieu-LDDD, p388).
- **Streaming** (`dma_map_single(dev, ptr, size, dir)` with `DMA_TO_DEVICE`/`FROM_DEVICE`/`BIDIRECTIONAL`): once mapped the "buffer … belongs to the device and not to the CPU anymore" (Madieu-Mastering, p560). The map call will "clean/invalidate the caches"; **the CPU must not touch it until `dma_unmap_*`** — correct pattern is to unmap in the completion IRQ, then "now it is safe to access the buffer" (Madieu-Mastering, p561, p563). If the CPU must peek mid-transfer, resync with `dma_sync_single_for_cpu/device()` (Madieu-Mastering, p565).
- **Scatter-gather:** build an `sg_table`, populate with `sg_set_buf()/sg_set_page()`, map with `dma_map_sg()`, iterate with the `for_each_sg()` macro; read per-entry bus address/length with `sg_dma_address(sg)` / `sg_dma_len(sg)` because "their location in the structure varies from one architecture to another" (Madieu-Mastering, p564).
- Design rule: **"use streaming mapping when you can and coherent mapping when you must"** — coherent is trivial to code but expensive (Madieu-Mastering, p565).
- Allocation-source facts that constrain DMA buffers: `kmalloc`/`kzalloc` memory is "guaranteed to be physically contiguous" and cacheline-aligned (good for small DMA buffers) but a single call caps at **4 MB** on x86/ARM (MAX_ORDER=11) (Billimoria-P1, p433, p442). For contiguous chunks over that 4 MB limit use **CMA** (Contiguous Memory Allocator), transparently wired into the DMA engine (Billimoria-P1, p499). RAM zones `DMA`/`DMA32`/`Normal` back DMA-capable allocations (Billimoria-P1, p389). `virt_to_phys()` is valid **only** on lowmem/`kmalloc` addresses — never on `vmalloc`/`ioremap` KVAs (Billimoria-P1, p393).
- Memory barriers matter for DMA descriptor rings: put a `wmb()` between descriptor-word writes so the device never sees a half-written descriptor (Billimoria-P2, p416–417) — see §5.
- **Zero-copy cross-driver sharing** is exactly the DMA-buf/PRIME path in §3; the exporter allocates DMA memory and hands out an fd another driver imports.

## 3. Char device / ioctl / file_operations (the shape a DRM UAPI takes)

_Our guest DRM driver exposes a `/dev/dri/*` char device; this is the ioctl/mmap surface it implements. Madieu's V4L2/DRM chapters give the closest real "GPU-buffer" UAPI._

- **Char-device init ordering:** `alloc_chrdev_region(&dev, 0, count, name)` (prefer dynamic over `register_chrdev_region`) → `class_create()` → per-minor `cdev_init(&cdev, &fops)`; set `cdev.owner = THIS_MODULE`; `cdev_add(&cdev, MKDEV(major, minor), 1)` → `device_create(class, NULL, devt, drvdata, fmt, …)` which "creates a device and registers it with Sysfs" (node appears in `/dev`, class in `/sys/class`) (Madieu-LDDD, p169–173). **Teardown reverses exactly:** `device_destroy` → `cdev_del` → `class_destroy` → `unregister_chrdev_region` (Madieu-LDDD, p173).
- `dev_t` = 12-bit major + 20-bit minor (`MINORBITS 20`); decode with `MAJOR()/MINOR()`, build with `MKDEV()` (Madieu-LDDD, p168). Legacy shortcut `register_chrdev(0, name, &fops)` returns a major in one call (Giometti-Cookbook, p85).
- `struct file_operations` members we implement: `open`, `release`, `read`, `write`, `llseek`, `poll`, `mmap`, `unlocked_ioctl` — "none of them are mandatory"; a missing `mmap` makes `mmap()` return `-ENODEV`, missing `read/write` returns `-EINVAL` (Madieu-LDDD, p165–166).
- **Per-open GPU context lives in `filp->private_data`** — set it in `open` (recover your device struct via `container_of(inode->i_cdev, struct mydev, cdev)`); it "will be available to other system calls invoked on the same file descriptor". `release` mirrors `open` and should free resources only on last close via a `users--` refcount (Madieu-LDDD, p175–177).
- User↔kernel copies: `copy_from_user`/`copy_to_user` return "the number of bytes that could not be copied" (0 = success), **may sleep** (so never under a spinlock — see §5), and the `__user` cookie marks pointers you must not deref (Madieu-LDDD, p173–174).
- **ioctl:** `long (*unlocked_ioctl)(struct file *, unsigned int cmd, unsigned long arg)` plus an optional `compat_ioctl` for 32-on-64 userspace (Madieu-Mastering, p327). Unknown command returns **`-ENOTTY`** (Madieu-LDDD, p167). Encode commands with `_IO`/`_IOR`/`_IOW`/`_IOWR(magic, nr, type)` — the direction and `sizeof(type)` are baked into the command number and the kernel uses the size "about the size to be copied" (Madieu-LDDD, p189–190). Pick one magic per driver, share the command header between kernel and user, "gather … arguments in a structure and just pass a pointer" (Madieu-LDDD, p190–192).
- Validate before dispatch: check `_IOC_TYPE(cmd)` against your magic and reject mismatches; `_IOC_NR/_IOC_SIZE/_IOC_DIR(cmd)` are also readable at runtime (Giometti-Cookbook, p220–223). (Neither driver book calls `access_ok()` — they rely on `copy_*_user`/`get_user`/`put_user`.)
- **The DRM-style "one struct of function-pointers per verb" pattern** is exactly V4L2's: a single dispatcher (`video_ioctl2`) "takes care of argument copying between the kernel and user space and performs some sanity checks", routing to an ops table (`v4l2_ioctl_ops`, ">120 entries") of typed per-command handlers like `int (*vidioc_reqbufs)(struct file*, void *fh, struct v4l2_requestbuffers *)`; individual verbs can be gated with `v4l2_disable_ioctl()` before device registration (Madieu-Mastering, p331–332).
- **The buffer-sharing UAPI infinigpu will mirror:** `VIDIOC_QUERYBUF` returns a buffer offset in `buf.m.offset`, then userspace does `mmap(NULL, len, …, fd, buf.m.offset)` (Madieu-Mastering, p449). Cross-driver zero-copy is `VIDIOC_EXPBUF` → returns a **dma-buf fd**; the importer sets `buf.memory = V4L2_MEMORY_DMABUF` + `buf.m.fd` (Madieu-Mastering, p450). And the exact GPU seam: **DRM allocates a buffer returning a GEM handle, `DRM_IOCTL_PRIME_HANDLE_TO_FD` exports it as a dma-buf fd through PRIME**, then `drmModeAddFB2()/drmModeSetPlane()` scan it out — "PRIME is the name of the drm dma-buf interface layer integrated with GEM" (Madieu-Mastering, p454). This is how our host arbiter hands a shared dma-buf to a DRM display/compositor.
- **mmap fop** `int (*mmap)(struct file *, struct vm_area_struct *vma)`: the kernel pre-builds the VMA (`vm_start`/`vm_end`/`vm_pgoff`/`vm_page_prot`/`vm_flags`) and your callback finishes it. Core call is `remap_pfn_range(vma, vma->vm_start, pfn, size, vma->vm_page_prot)` which "maps physical contiguous memory into a process address space" (Madieu-LDDD, p376, p381). PFN source: `kmalloc/__get_free_pages` → `virt_to_phys(addr) >> PAGE_SHIFT`; `alloc_pages` → `page_to_pfn`; `vmalloc` → `vmalloc_to_pfn`; **device MMIO/BAR** → use `io_remap_pfn_range()` (or `vm_iomap_memory(vma,start,len)`) with PFN from the ioremap physical address (Madieu-LDDD, p377, p379). OR the right flags onto `vm_flags`: `VM_IO`, `VM_PFNMAP` ("a page range managed without a backing struct page, just pure PFN"), `VM_DONTCOPY/DONTEXPAND/DONTDUMP` (Madieu-LDDD, p377).
- **mmap pitfalls:** honor the userspace offset `vma->vm_pgoff << PAGE_SHIFT`; mappings must be `PAGE_SIZE`-aligned multiples; you cannot `remap_pfn_range` a raw `kmalloc` buffer safely ("you might corrupt slab internal data structures" — allocate whole pages / `kzalloc(PAGE_SIZE)` instead) (Madieu-LDDD, p378; Giometti-Cookbook, p224). For device-visible/coherent mappings set `vma->vm_page_prot = pgprot_noncached(vma->vm_page_prot)` — the author saw latency drop from ~20 ms to <200 µs (Madieu-LDDD, p380). Post-mmap, userspace touches the buffer with **zero further syscalls** — the reason to expose GPU buffers this way (Giometti-Cookbook, p229).
- **sysfs for device presence/control signaling:** `DEVICE_ATTR_RW/RO/WO(name)` generates `dev_attr_<name>` with `show`/`store` callbacks (retrieve private struct via `dev_get_drvdata(dev)`); attach a whole group at class-init with `class->dev_groups`. For poll/select on an attribute, the driver **must call `sysfs_notify(kobj, dir, attr)`** after updating — else "any polling process will end up waiting indefinitely" (poll returns `POLLERR|POLLPRI`) (Madieu-LDDD, p520–524; Giometti-Cookbook, p269–271).

## 4. Interrupts / MSI-X (doorbells & completion signaling)

_How our virtual device raises completions to the guest, and how the guest driver services them without stalling the vCPU._

- **MSI is literally a doorbell write:** the device signals by "write[ing] a small amount of interrupt-describing data to a special memory-mapped I/O address" (Madieu-Mastering, p529). Key limitation to design around: the MSI data word only selects *which vector* fires — "that data is not available for the device to communicate additional information to the interrupt handler" (Madieu-Mastering, p529). So completion payload must live in a shared ring/BAR, not the interrupt itself.
- MSI = 1–32 vectors sharing one address; **MSI-X scales to 2,048** with per-vector address+data pairs, and makes 64-bit addressing + per-vector masking mandatory (Madieu-Mastering, p531). Allocate generically with `pci_alloc_irq_vectors(dev, min, max, flags)` (flags `PCI_IRQ_MSIX`/`MSI`/`LEGACY`/`ALL_TYPES`/`AFFINITY`; ALL_TYPES tries "MSI-X mode … first"), returns count or `-ENOSPC` (Madieu-Mastering, p550–551). Map vector→Linux IRQ with `pci_irq_vector(dev, nr)`; **this must run before `request_irq()`**, and MSI(-X) vs legacy are "mutually exclusive" (Madieu-Mastering, p536, p552).
- Handler prototype `irqreturn_t handler(int irq, void *dev)`; register with `request_irq()` / managed `devm_request_irq()` (four APIs, "devm_request_threaded_irq() (recommended!)") (Madieu-Mastering, p64; Billimoria-P2, p154). The `dev` cookie is your per-device context and "cannot be NULL for shared IRQs"; **PCI devices need `IRQF_SHARED`**, so the top half must check "is this my interrupt?" and return `IRQ_NONE` if not, else `IRQ_HANDLED`, or `IRQ_WAKE_THREAD` to defer (Madieu-Mastering, p65–69; Billimoria-P2, p172).
- **Top half runs "in a so-called hard-IRQ context, with interrupts disabled"** and is atomic: no sleeping, no `copy_*_user`, and any allocation "must use the GFP_ATOMIC flag" (Madieu-Mastering, p72; Billimoria-P2, p165). While it runs the IRQ is masked so the handler is "inherently reentrant-safe" (Billimoria-P2, p169). Keep it to tens of microseconds; consistently >100 µs → move to a bottom half / threaded handler (Billimoria-P2, p170, p213).
- **Threaded IRQs** (`request_threaded_irq(irq, top, thread_fn, flags, name, dev)`): top half returns `IRQ_WAKE_THREAD`, `thread_fn` "runs as a kernel thread in process context" (can sleep, do blocking work). If the top half is `NULL`, `IRQF_ONESHOT` is mandatory (Madieu-Mastering, p74–76; Billimoria-P2, p182, p189).
- **Waking a blocked submitter from the handler is legal** because "the wake_up*() APIs are non-blocking" — the canonical submit-then-signal-completion pattern: the read/ioctl path sleeps on `wait_event*()`, the completion IRQ calls `wake_up` (Billimoria-P2, p164, p168).
- Deferred-work choices: **tasklet** is "strictly serialized wrt itself" (never concurrent with itself), built on softirq, `tasklet_init()`+`tasklet_schedule()`; a **softirq** vector "can indeed run in parallel with itself on another CPU"; a **workqueue** callback runs "in a preemptible process context" and may block/do I/O (`INIT_WORK`+`schedule_work`, cancel with `cancel_work_sync`) (Billimoria-P2, p202, p212, p281, p287).
- Level-triggered lines must be acked in the handler or the CPU "will signal the interrupt again immediately" (Madieu-Mastering, p65). Data shared between a hard-IRQ and a process-context path needs `spin_lock_irqsave()` (see §5) (Madieu-Mastering, p86).

## 5. Kernel memory allocation & synchronization (guest driver + arbiter concurrency)

_Allocation choices in the guest driver, and the locking discipline the multi-VM arbiter must reason about._

- **GFP flag rule:** process context that may sleep → `GFP_KERNEL`; any atomic/interrupt context or while holding a spinlock → `GFP_ATOMIC` (else deadlock/crash) (Billimoria-P1, p410). `kzalloc` is "the preferred and recommended API" (zeroes the slab → avoids info-leak) (Billimoria-P1, p434).
- **Allocator selection:** everything ultimately flows through the buddy/page allocator (Billimoria-P1, p398). `__get_free_pages(gfp, order)` returns `2^order` physically-contiguous pages; `alloc_pages()` returns a `struct page *` (convert via `page_address()`) (Billimoria-P1, p409). Largest single contiguous chunk = `2^(MAX_ORDER-1)` = **4 MB** (MAX_ORDER=11 on x86/ARM) (Billimoria-P1, p400). `kmalloc` uses `kmalloc-'n'` slab caches only up to **8,192 bytes**, larger goes straight to the page allocator, and single-call caps at **4 MB** (Billimoria-P1, p442, p446).
- `vmalloc` gives virtually-contiguous, page-aligned memory with **no physical-contiguity guarantee**, is demand-paged ("No physical memory is actually allocated at this time"), and **must run in process context only** (may sleep) — so never for DMA or in IRQ context (Billimoria-P1, p480–485). `kvmalloc(size, GFP_KERNEL)` tries kmalloc then "falls back to … vmalloc()" — the right call when size is large/unknown (Billimoria-P1, p487). Decision: kmalloc for atomic-context + DMA + small; vmalloc for large software buffers only (Billimoria-P1, p494).
- **Spinlock vs mutex (the core arbiter decision):** critical section in atomic/interrupt context or that "cannot sleep? Use the spinlock"; process context that may sleep? "Use the mutex lock" (Billimoria-P2, p321). Mutex losers sleep (~2 context switches); spinlock losers spin (cheap for short sections) (Billimoria-P2, p320). Mutex: only the owner unlocks, no recursion, and "**mutexes may not be used in … interrupt contexts**" (Billimoria-P2, p324). **Never sleep inside a spinlock** — `copy_to_user()` can sleep, so a section that copies to user must use a mutex (Billimoria-P2, p342).
- IRQ-safe locking: when a hard-IRQ handler and a process path share data, use `spin_lock_irqsave()/spin_unlock_irqrestore()` — it "save[s] and restore[s] the interrupt mask" on the local core (safer than plain `spin_lock_irq()` which unconditionally re-enables); the **same** lock must be taken in both paths (Billimoria-P2, p354–358).
- Lock-free / low-overhead tools: `atomic_t`/`atomic64_t` (indivisible int ops, always via the accessors); newer `refcount_t` for reference counters (saturating, UAF/overflow-protected) — good for per-VM/per-context refcounts; `set_bit/clear_bit/test_and_set_bit` for atomic RMW on device-register bits across all cores; per-CPU vars (`get_cpu_var`/`put_cpu_var`, disables preemption) (Billimoria-P2, p362, p368, p371, p389). Reader-writer spinlocks exist but "writers can starve" — avoid unless the read section is long (Billimoria-P2, p381).
- **Memory ordering:** `rmb()/wmb()/mb()` suppress CPU/compiler reordering — needed for DMA descriptor setup (`wmb()` between descriptor-word writes, per the real Realtek 8139 driver); `READ_ONCE()/WRITE_ONCE()` force exactly one access, and note "atomicity is a separate construct, not guaranteed by using the volatile keyword" (Billimoria-P2, p416–418). `false sharing` (two hot vars on one 64-byte cacheline written by different cores) causes cache ping-pong — pad them apart (Billimoria-P2, p384).
- **Completion primitive for command submit/ack:** the kernel condition-variable is the *completion* mechanism ("Its equivalent … is called the completion mechanism") — a submitter blocks on `wait_for_completion()`, the worker/IRQ calls `complete()`; used internally by `kthread_stop()` (Billimoria-P2, p282, p382). A counting semaphore (`down()/up()`, no ownership) can "signal another waiting task that a certain milestone has been reached" for producer/consumer rings (Billimoria-P2, p336).
- Debugging aids to enable in dev builds: **lockdep** (`CONFIG_PROVE_LOCKING`) gives runtime "proof of locking correctness" (catches AB-BA); `CONFIG_DEBUG_ATOMIC_SLEEP` loudly reports sleeping-in-atomic — a non-debug kernel "silently lets the same bug pass" (Billimoria-P2, p348, p400). Enforce a documented lock-ordering to avoid AB-BA deadlock (Billimoria-P2, p325).

## 6. Vulkan: queues, command buffers, submission, sync, memory, external-memory, present

_The host renderer replays this stream; the guest encoder serializes it. Timeline semaphores + external memory are the load-bearing pieces for many-VM sharing and zero-copy._

**Queues & submission.** A queue is "the entry point to submit commands", and **submission is single-threaded** — "Submission using a queue is a single-threaded operation" (Vulkan-Mastering, p85). Queue-family flags: `GRAPHICS_BIT` (draws), `COMPUTE_BIT` (dispatch/traceRays), `TRANSFER_BIT` (copies, which "use direct memory access (DMA) to speed up" CPU↔GPU transfer); there is always ≥1 family that does everything (Vulkan-Mastering, p86–87). Queues are created *at device creation* (`vkCreateDevice` + `pQueueCreateInfos`) and fetched afterward with `vkGetDeviceQueue(device, family, index, &queue)` (Vulkan-Mastering, p89; RenderingCookbook, p321). GPU execution order follows "the order of the array submitted to the … queue" (Vulkan-Mastering, p97). `VkSubmitInfo` fields we must serialize: `pWaitSemaphores`/`pWaitDstStageMask`/`pCommandBuffers`/`pSignalSemaphores` (RenderingCookbook, p192).

**Command pools/buffers (the wire-encoding shape).** A command pool binds to one queue family (`VkCommandPoolCreateInfo.queueFamilyIndex`), create with `VK_COMMAND_POOL_CREATE_RESET_COMMAND_BUFFER_BIT`; buffers via `vkAllocateCommandBuffers` at `..._LEVEL_PRIMARY` (Vulkan-Mastering, p91). **Pools are not thread-safe** — "any kind of pool needs to be externally synchronized by the user", so use **one pool per thread × per frame-in-flight** (F×T), and "Simultaneous recording on the same CommandPool is forbidden" (Vulkan-Mastering, p97, p99). Reuse by `vkResetCommandPool` at frame start ("much more efficient on CPU time") rather than freeing (Vulkan-Mastering, p98). Record lifecycle: `vkBeginCommandBuffer` (flag `ONE_TIME_SUBMIT_BIT` for transient, `SIMULTANEOUS_USE_BIT` for re-submitted) → `vkCmdBeginRenderPass` → bind pipeline/descriptors → `vkCmdDraw` → `vkCmdEndRenderPass` → `vkEndCommandBuffer` (RenderingCookbook, p151–156). Primary buffers are "the only one that can be submitted to a queue"; secondaries inherit only RenderPass+FrameBuffer (so "viewport and stencil states must be set again") and merge via `vkCmdExecuteCommands` (Vulkan-Mastering, p98–100). **Commands can be pure data**: indirect draws are just "an array of VkDrawIndirectCommand structures" consumed at a stride — a natural template for serializing draws over our transport (RenderingCookbook, p274, p278).

**Synchronization (highest-value section for us).** **Fences** are host↔device: "Fences must be used to ensure the GPU has finished" a frame's resources; one per frame-in-flight, created signaled (`VK_FENCE_CREATE_SIGNALED_BIT`) so frame 0 proceeds, reset with `vkResetFences` (Vulkan-Mastering, p92, p129). **Timeline semaphores** are the better multiplexing primitive: a monotonically increasing 64-bit value waitable by **both CPU and GPU** (collapses fence+binary-semaphore into one), promoted to core in 1.2 — create by chaining `VkSemaphoreTypeCreateInfo{ semaphoreType = VK_SEMAPHORE_TYPE_TIMELINE }` into `pNext`; CPU-wait via `vkWaitSemaphores` (timeout in **nanoseconds**) (Vulkan-Mastering, p130–133). **Correctness rules:** "the signal value … is always increased" (resubmitting an equal/smaller value is a validation error); never wait a value only signaled within the same submission or "we would deadlock the GPU" (Vulkan-Mastering, p134). Cross-queue (compute→graphics) sync: the producer signals its timeline value, the consumer adds that semaphore to its wait list at the right stage — and "Using the right wait stage is important" (e.g. `VERTEX_ATTRIBUTE_INPUT` for compute-written vertices, `FRAGMENT_SHADER` if compute wrote a texture) (Vulkan-Mastering, p137–138). Prefer the `VK_KHR_synchronization2` submit path (`VkSubmitInfo2KHR` + `VkSemaphoreSubmitInfoKHR`) (Vulkan-Mastering, p133). **Pipeline barriers / layout transitions:** fill `VkImageMemoryBarrier` (`srcAccessMask`/`dstAccessMask`/`oldLayout`/`newLayout`/`srcQueueFamilyIndex`/`dstQueueFamilyIndex`/`image`/`subresourceRange`) and issue `vkCmdPipelineBarrier(cmd, srcStage, dstStage, …)`; e.g. `UNDEFINED→TRANSFER_DST` is `0→TRANSFER_WRITE` across `TOP_OF_PIPE→TRANSFER` (RenderingCookbook, p162–163).

**Memory.** Pick a heap by reading `vkGetPhysicalDeviceMemoryProperties`, then matching the `typeFilter` bit AND the required `VkMemoryPropertyFlags` (RenderingCookbook, p153). Buffer flow: `vkCreateBuffer` → `vkGetBufferMemoryRequirements` → `vkAllocateMemory` (`memoryTypeIndex`) → `vkBindBufferMemory(dev, buf, mem, 0)`; images use `vkBindImageMemory` after `vkGetImageMemoryRequirements` (RenderingCookbook, p154, p158). CPU-updated data → `HOST_VISIBLE | HOST_COHERENT` written via `vkMapMemory`/memcpy; GPU-resident → `DEVICE_LOCAL` ("can't be accessed directly from the host"), filled through a **staging buffer** (`HOST_VISIBLE` `TRANSFER_SRC` → `vkCmdCopyBufferToImage`/`vkCmdCopyBuffer`) (RenderingCookbook, p157, p166). Keep a large staging buffer persistently mapped (`VMA_ALLOCATION_CREATE_MAPPED_BIT`); mind image-upload alignment (round to a texture alignment before placing in the staging buffer) (Vulkan-Mastering, p92, p94). Images must be created `initialLayout = VK_IMAGE_LAYOUT_UNDEFINED` (RenderingCookbook, p159).

**External / shared memory (dma-buf) — GAP + adjacent facts.** Neither Vulkan book covers `VK_KHR_external_memory`, `VK_EXT_external_memory_dma_buf`, `VkExportMemoryAllocateInfo`, `VkImportMemoryFdInfoKHR`, `vkGetMemoryFdKHR`, or external semaphores — **go to the Vulkan spec for the export/import contract**. Closest usable mechanics here: resources default to `sharingMode = VK_SHARING_MODE_EXCLUSIVE` (one queue family owns at a time), so an exported/shared image must either use `CONCURRENT` or do explicit ownership transfer (RenderingCookbook, p154). **Queue-family ownership transfer** (needed whenever a resource crosses queues/processes) is a **release/acquire barrier pair**: on the source queue a barrier with `srcQueueFamilyIndex=producer, dstQueueFamilyIndex=consumer`, and a matching one on the consumer — "another memory barrier that will also transfer ownership"; use `VK_QUEUE_FAMILY_IGNORED` when no transfer is intended (Vulkan-Mastering, p96; RenderingCookbook, p162). The exported surface itself is built like an offscreen color target (`SAMPLED | COLOR_ATTACHMENT` usage, given a fixed layout at creation so the consumer sees a stable layout) — the same image machinery, just backed by imported/exported `VkDeviceMemory` (RenderingCookbook, p433).

**Swapchain / present / headless.** A swapchain is "a queue of rendered images", and crucially each frame "is rendered as an offscreen image" first — **so a headless host replay stops at that offscreen image and skips the swapchain entirely** (RenderingCookbook, p139). Present-mode policy: prefer `MAILBOX_KHR` (no tearing), fall back to the always-supported `FIFO_KHR`; request `minImageCount + 1` (RenderingCookbook, p141). `VkSwapchainCreateInfoKHR.imageUsage = COLOR_ATTACHMENT_BIT | TRANSFER_DST_BIT` (RenderingCookbook, p142). Present loop: `vkAcquireNextImageKHR(dev, swapchain, timeout, acquireSem, fence, &idx)` → submit (wait `acquireSem` at `COLOR_ATTACHMENT_OUTPUT`, signal `renderSem`) → `vkQueuePresentKHR` (waits `renderSem`) — the **two-semaphore pattern** ensures "the rendering process waits for the swap chain image to become available" and presentation waits for render (RenderingCookbook, p149, p191–192). The final pass normally transitions to `VK_IMAGE_LAYOUT_PRESENT_SRC_KHR`; **for headless export, transition to `TRANSFER_SRC`/your consumer layout and never touch `PRESENT_SRC`**, allocating `DEVICE_LOCAL` offscreen targets with explicit usage flags via `createImage` (RenderingCookbook, p201–202, p434).

## 7. KVM/QEMU device model, virtio, vhost, guest memory (the host platform)

_The platform our virtual GPU device plugs into. vhost-user is the architectural cousin of our out-of-process vfio-user server (the books cover its in-kernel ancestor vhost_net)._

- **KVM control surface = three ioctl-fd tiers:** system ioctls on `/dev/kvm` (`KVM_CREATE_VM`, `KVM_CHECK_EXTENSION`) → VM ioctls on `vmfd` (`KVM_CREATE_VCPU`, memory layout) → vCPU ioctls on the vcpu fd (`KVM_RUN`) (KVM-Mastering, p75). QEMU is the only process touching the device file — it "opens `/dev/kvm`" and drives everything through `ioctl()`; libvirt just spawns "a separate QEMU-KVM process" per VM (KVM-Mastering, p61, p68).
- **vCPU run loop:** each vCPU is a POSIX thread that loops on `ioctl(KVM_RUN)`; guest code runs directly on the host CPU and returns on a **VM exit** — a switch on `run->exit_reason` (`KVM_EXIT_IO`, `KVM_EXIT_MMIO`, `KVM_EXIT_SHUTDOWN`, …). When the guest touches a QEMU-emulated register, "KVM has to exit back to QEMU and pass control" (KVM-Mastering, p70, p82). The vcpu fd is `mmap`'d to share the `kvm_run` struct with the kernel (KVM-Mastering, p69). Hardware needs VT-x/AMD-V (VMX root vs non-root, VMCS per vCPU) and **EPT/NPT** (SLAT) — without it "the hypervisor must exit the virtual machine to perform address translations" (KVM-Mastering, p74).
- **Guest memory (key for out-of-process DMA):** "the physical RAM of the guest is inside the QEMU process address space" — QEMU "emulate[s] the virtual machine's physical address space within the user-mode address space of QEMU-KVM", registered into KVM with `KVM_SET_USER_MEMORY_REGION` (KVM-Mastering, p63, p67, p71). An out-of-process device must map/share that region to DMA into guest RAM. For a DMA-heavy device, back guest RAM with **hugepages** (`<memoryBacking><hugepages>`, `nr_hugepages`) and mind **NUMA node locality** for the passed device (KVM-Mastering, p602, p614).
- **QEMU concurrency & device split:** QEMU is "multithreaded, event-driven (with a big lock)" — a main event loop + I/O worker threads + one thread per vCPU; device code splits host-backend vs emulation ("hw/block/ contains the code for device emulation") — the same split a PCI GPU model follows (KVM-Mastering, p71, p72).
- **virtio** is the paravirtual I/O framework: guest frontend driver + hypervisor backend over a **virtqueue + vring**, where "the memory is mapped between QEMU and the guest" (the shared-ring model a fast virtual GPU reuses) (KVM-Mastering, p619–620). An in-tree **3D GPU precedent already exists**: `virtio-vga`/`virtio-gpu`, "A paravirtual 3D virtual graphics card that is based on the virgl project" (KVM-Mastering, p204). Windows guests need `virtio-win` drivers loaded at install (KVM-Mastering, p371). Wire format (split/packed rings, feature bits) is only in the linked Virtio 1.1 spec (KVM-Mastering, p633).
- **vhost = move the device backend out of the QEMU vCPU thread** — the exact architectural move our out-of-process device makes. `vhost_net` "provides in-kernel virtio devices for KVM" and "vhost is a common framework that can be used by different drivers"; it's transparent to the guest ("there is no change to the frontend driver"), reduces copies/latency, is configured over ioctls (`-netdev tap,vhost=on`) through a host char device `/dev/vhost-net`, and negotiates features between QEMU and the backend — model this handshake for our device (KVM-Mastering, p124, p624–626). (Books cover in-kernel vhost, not `vhost-user`/`vfio-user` by name — but it's the same offload pattern.)
- **The existing GPU-sharing hardware paths (why infinigpu exists):** IOMMU (VT-d/AMD-Vi) lets a VM "access peripheral hardware directly (graphics cards, …)"; **full PCI passthrough** binds the card to `vfio-pci` and hands the whole GPU to one VM (best perf, no density) (KVM-Mastering, p45, p138, p210, p212). **SR-IOV** shares one device via Virtual Functions but needs hardware+driver support (KVM-Mastering, p45). **Mediated devices (mdev) = the vGPU software-partitioning path**: NVIDIA vGPU loads `nvidia_vgpu_vfio`/`mdev`/`vfio_mdev`/`vfio_iommu_type1` and a VM claims a slice via `<hostdev mode='subsystem' type='mdev' model='vfio-pci'>` keyed by a UUID — "the shared graphics card scenario" and the closest existing pattern to a synthesized virtual GPU (KVM-Mastering, p209, p212). Our design forgoes all of these (licensing / no-partition A5000) in favor of userspace API-remoting (see `../research/06-data-plane-and-host-gpu.md`).

## 8. Rust systems idioms for the no_std guest side + FFI host code

_Writing safe Rust over C hypervisor/kernel FFI, and the concurrency primitives for the arbiter._

- **`unsafe {}` permits exactly:** dereferencing raw pointers (`*const T`/`*mut T`), calling an unsafe/`extern` fn, reading a `union` field, and mutating a `static mut` — and "the borrow checker is still active" inside it (Rust-Reference, p372). Raw pointers are the FFI currency: they can be null and need `unsafe` to deref; you can cast `&T as *const T` but "can't cast a &T to a *mut T" (Rust-Reference, p210–211).
- **Calling C (QEMU/libvirt/kernel UAPI):** declare in `extern "C" { … }` (the `"C"` ABI is cdecl and the default) and call inside `unsafe`; the recommended shape is to **hide the `unsafe` behind a safe wrapper returning `Result`/`Option`** (Rust-Reference, p380–381). For a large C header surface, generate bindings with **`bindgen` in `build.rs`** (such crates conventionally take a `-sys` suffix); FFI is "zero-cost … a minimal runtime cost is incurred when linking" (Rust-Reference, p385–387, p396).
- **Being called by C** (a Rust device callback into C QEMU): `#[no_mangle] pub extern "C" fn`, `crate-type = ["cdylib"]`, and note you "cannot pass Rust-specific data types to the FFI boundary" (no `Option`/`Result`/`&dyn` across it) (Rust-Reference, p384). Also: "trait objects … are fat pointers double the size of a normal pointer" (16 vs 8 bytes) — never hand a `&dyn Trait` where C expects a plain pointer (Rust-Reference, p304).
- **Matching C ABI for ioctl structs:** use `#[repr(C)]`; a `#[repr(C)] union` "maps directly to a C union", is Copy-only and "unsafe to read from" (the idiom for `kvm_run`-style union payloads); the compiler "insert[s] padding between … struct fields", so verify with `std::mem::size_of`; use `std::os::raw` types (`c_char`, `c_uint`) and `CString`/`CStr` for C strings (Rust-Reference, p289, p303, p380).
- **Resource safety:** `Box<T>` is "the simplest way to allocate values in the heap" and owns the value; `Drop` is the RAII destructor (runs at end of scope, LIFO, "inspired by the RAII principle of C++") — ideal for auto-closing fds / QMP sockets / qcow2 handles (Rust-Reference, p212, p214).
- **Concurrency for the arbiter:** `Rc<T>` is non-atomic single-thread refcount; **use `Arc<T>` for anything shared across worker threads** (atomic) (Rust-Reference, p213, p217). `Cell`/`RefCell` move borrow checks to runtime (panic `BorrowMutError` on aliasing) and "are not thread safe" — can't cross the vCPU/iothread boundary (Rust-Reference, p225). `thread::spawn` requires `F: Send + 'static`; `move` closures transfer ownership in (Rust-Reference, p319, p330). Shared mutable device state = `Arc<Mutex<T>>` (`lock()` blocks); `RwLock` for read-heavy state but on Linux it "suffers from the writer starvation problem" (Rust-Reference, p324, p327).
- **`Send`/`Sync` are `unsafe` auto traits** — "If something is Sync then a reference to it … &T is Send". Hand-implementing them for a raw C device handle is a deliberate safety assertion; the book's cautionary example is `Rc` opting *out* to avoid "the wrong reference counts … early frees and dangling pointers" — exactly the hazard when a raw C pointer must not cross threads (Rust-Reference, p332, p376).
- **no_std / kernel relevance:** the book has no dedicated no_std chapter, but flags that "when doing low-level embedded system development or any program that interfaces with the operating system kernel, you need to switch to unsafe mode" (Rust-Reference, p372). For the guest DRM/WDDM driver side, the concrete no_std/kernel-FFI patterns (allocator, panic handler, `#[repr(C)]` UAPI structs) will need a kernel-Rust source beyond this corpus.
